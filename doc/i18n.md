# i18n

`Etyon` 现在通过共享包 `@etyon/i18n` 提供统一的国际化能力，供 `Electron main`、`Electron renderer` 和 `CLI` 共同使用。

其中 `renderer` 层已经切到基于 `i18next` + `react-i18next` 的实例模型：`I18nProvider` 内部持有独立的 `i18next instance`，`useI18n()` 是对官方 `useTranslation()` 的轻量封装。`main` 和 `CLI` 继续使用无 React 的 `createTranslator()`，但底层和 `renderer` 共享同一套资源与初始化选项。

## 支持语言

- `en-US`
- `zh-CN`
- `ja-JP`

应用默认语言为 `en-US`。`system` 不是实际语言，而是一个“跟随系统”的偏好值。

## 包结构

| 路径                                               | 说明                                                                                      |
| -------------------------------------------------- | ----------------------------------------------------------------------------------------- |
| `packages/i18n/src/index.ts`                       | 共享的 locale schema、解析逻辑、`i18next` 初始化选项、headless 翻译器工厂、`CLI` 参数解析 |
| `packages/i18n/src/react.tsx`                      | `I18nextProvider` 封装、`useI18n()` hook、React 实例创建                                  |
| `packages/i18n/src/locales/en-US/translation.json` | 英文基准文案                                                                              |
| `packages/i18n/src/locales/zh-CN/translation.json` | 简体中文文案                                                                              |
| `packages/i18n/src/locales/ja-JP/translation.json` | 日文文案                                                                                  |
| `packages/i18n/src/i18next.d.ts`                   | `i18next` 类型增强，给 key 与资源结构提供类型信息                                         |
| `.vscode/i18n-ally-custom-framework.yml`           | 让 `i18n Ally` 识别 `useI18n({ keyPrefix })` 的自定义框架配置                             |

## 设计说明

- `renderer` 使用官方推荐的 `I18nextProvider` + `useTranslation()` 组合，因此语言切换会通过 `languageChanged` 事件自动触发重渲染。
- 由于当前翻译资源都随包静态内置，不走异步加载，所以初始化时显式关闭了 `Suspense`，避免首屏额外引入 loading 边界。
- 翻译资源已经改为标准的嵌套 JSON 结构，因此 `i18next` 使用默认的点分隔 key 解析规则，例如 `settings.common.save` 会映射到 `settings.common.save` 路径。
- `@etyon/i18n` 会在初始化前统一标准化插值占位符，因此资源文件里的 `{name}` 和 `{{name}}` 两种写法都会被转换成 `i18next` 可识别的插值格式；业务层统一使用 `t("key", { name: value })`。
- `useI18n()` 仍然保留，原因是业务层可以继续通过统一入口获取 `t`、`ready`、`locale` 和 `changeLanguage()`，同时内部实现完全以 `i18next` 为主。
- `.vscode/settings.json` 与 `.vscode/i18n-ally-custom-framework.yml` 已经配套更新，使 `i18n Ally` 可以直接识别 `packages/i18n/src/locales` 下的资源文件，以及 `useI18n({ keyPrefix: "..." })` 的作用域。

## Locale 解析规则

### Desktop

`desktop` 会把用户偏好持久化到 `settings.locale`。运行时解析顺序如下：

1. 如果 `settings.locale !== "system"`，直接使用该值
2. 否则读取系统 locale
3. 精确匹配支持语言
4. 如果只有语言前缀可用，则回退到：
   `en -> en-US`、`zh -> zh-CN`、`ja -> ja-JP`
5. 如果仍然无法识别，回退到 `en-US`

`Electron main` 使用 `app.getLocale()`，`renderer` 使用 `navigator.languages` / `navigator.language`，两端共享同一套 `resolveLocale()` 规则。

### CLI

`CLI` 的解析顺序如下：

1. `--lang <locale>`
2. `-l <locale>`
3. `--lang=<locale>`
4. `LC_ALL`
5. `LC_MESSAGES`
6. `LANG`
7. `en-US`

如果显式传入了不支持的 locale，`CLI` 会输出受控错误并以退出码 `1` 结束。

## 使用方式

### Headless API

```ts
import { createTranslator, resolveLocale } from "@etyon/i18n"

const locale = resolveLocale("system", "zh_CN.UTF-8")
const { t } = createTranslator(locale)

t("settings.common.save")
```

### React API

```tsx
import { I18nProvider, useI18n } from "@etyon/i18n/react"

const Example = () => {
  const { t } = useI18n()

  return <button>{t("settings.common.save")}</button>
}
```

```tsx
import { useI18n } from "@etyon/i18n/react"

const LanguageSection = () => {
  const { t } = useI18n({ keyPrefix: "settings.language" })

  return <h2>{t("title")}</h2>
}
```

## 新增文案或语言

1. 在 `packages/i18n/src/locales/en-US/translation.json` 中新增嵌套 key
2. 在 `packages/i18n/src/locales/zh-CN/translation.json` 与 `packages/i18n/src/locales/ja-JP/translation.json` 中补齐同名路径
3. 如果是新语言，补充 locale 目录、`SUPPORTED_LOCALES`、`LocaleSchema` 和回退规则
4. 如果涉及 `desktop` 原生 UI，记得同时检查菜单和窗口标题
5. 改完后建议在 VS Code 里通过 `i18n Ally` 检查缺失 key 和翻译同步情况
